# ПРАВИЛА ПОСТАНОВКИ ЗАДАЧИ В JIRA
**Gentel Reminder:** В нашей команде действуют небольшие правила постановки задач. Отклонения допускаются только при личном согласовании с *разработчиком* на которого ставится задача и *Tech Lead/Team Lead/PM/Старший BA*

## EPIC
Доку для Front/Back прикрепляем в эпик, если эпик *весь (или почти весь) для одной команды*. В противном случае крепим артефакты Back и Front к отдельным таскам внутри эпика.
|       | Описание |
|-------|-------|
| Заголовок | Экран + результат |
| Цель | Одно предложение не более *15 слов*, которое отвечает на вопрос "Зачем это нужно?" |
| Приоритет | Выбираем конкретный из: *Blocker, Critical, High, Medium, Low* |
| Стейкхолдеры | Полный список людей, которых касается эпик |
| Шаги | Basic Flow, Alternative flow, Axeption flow |
| Результат | DoD, Acceptance criteria, KPI (если применимо) |
| Время | Story Point, оговоренная дата закрытия (если возможно) |
| Ссылки | Эпик описывающий реализацию как она есть сейчас (если имеется), ссылки на страницы (для Front) |
| Артефакты | *Activity Diagram (**всегда**)*, макеты (всегда для Front), ER-Diagram (если изменения в БД для Back), Sequense Diagram (если задействовано 2 и более сервиса), Class Diagram (если вносятся изменения в cписок атрибутов/методов) |

## TASK
|       | Описание |
|-------|-------|
| Заголовок | Обект действия + действие |
| Тип задачи | Выбираем конкретный из: (Front/Back) Development, Code Review, Deployment, Documentation |
| Описание | Описание *границ* таски в рамках всего эпика (если таска составная строго пунктами) |
| Приоритет | Выбираем конкретный из: *Blocker, Critical, High, Medium, Low* |
| Блокеры | Ссылки на блокирующие таски (если есть) |
| Результат | Из Acceptance Criteria выделены только связанные с таской |
| Время | Story Point, оговоренная дата выполнения (если возможно) |
| Валидация | Кто конкретно занимается тестированием |
| Артефакты | Спецификация API, эндпоинты (*если есть изменения*), дока с ручными тестами (если необходима) |

## BUG
Баги ставим в тот эпик к которому они относятся, если эпик существует и еще открыт. Если эпик закрыт или не описан стандартным способом, то баг ни к чему не прикрепляем.
|       | Описание |
|-------|-------|
| Заголовок | Страница + обект + поломка|
| Приоритет | Выбираем конкретный из: *Blocker, Critical, High, Medium, Low* |
| Шаги воспроизведения | Use Case этого flow от начала до ошибки |
| Результат | Ожидаемый и фактический результаты (с выделенными различиями) |
| Среда | UserID на котором зафиксирован баг, версия, ОС, логи *(при необходимости)* |
| Критерии приемки | Из AC Use Case *дублируем* те критерии, которые перестали выполняться |
| Артефакты | Из таски *дублируем* Activity Diagram и макеты (для Front) |
